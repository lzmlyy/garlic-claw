# 后端模型调用接口说明

本文档说明 Garlic Claw 当前后端如何配置 provider、如何统一调用模型，以及聊天接口、后端内部扩展调用、停止 / 重试 / 图片链路。

## 1. provider 配置入口

### 1.1 配置文件

AI provider 不再从环境变量读取，统一放在：

```text
config/ai-settings.json
```

建议从模板复制：

```bash
cp config/ai-settings.example.json config/ai-settings.json
```

### 1.2 支持的 provider 类型

当前只支持两类 provider：

- 官方 provider
  - `mode: "official"`
  - `driver` 直接使用官方实现
- 兼容 provider
  - `mode: "compatible"`
  - `driver` 只允许：
    - `openai`
    - `anthropic`
    - `gemini`

### 1.3 官方 SDK 映射

当前真实运行映射如下：

- `openai` -> `@ai-sdk/openai`
- `anthropic` -> `@ai-sdk/anthropic`
- `gemini` -> `@ai-sdk/google`
- `groq` -> `@ai-sdk/groq`
- `xai` -> `@ai-sdk/xai`
- `mistral` -> `@ai-sdk/mistral`
- `cohere` -> `@ai-sdk/cohere`
- `cerebras` -> `@ai-sdk/cerebras`
- `deepinfra` -> `@ai-sdk/deepinfra`
- `togetherai` -> `@ai-sdk/togetherai`
- `perplexity` -> `@ai-sdk/perplexity`
- `gateway` -> `@ai-sdk/gateway`
- `vercel` -> `@ai-sdk/vercel`
- `openrouter` -> `@openrouter/ai-sdk-provider`

兼容 provider 会复用对应 driver 的协议适配，其中 `openai` 兼容模式固定走 `/v1/chat/completions`。

## 2. 统一聊天入口

后端统一对外暴露聊天接口：

```text
POST /api/chat/conversations/:id/messages
```

这个接口负责：

- 读取 `provider` / `model`
- 读取模型能力配置
- 处理图片输入与 Vision Fallback
- 创建用户消息和 assistant 占位消息
- 以 SSE 增量返回生成过程

### 2.1 请求体

```json
{
  "content": "请分析下面这段 SQL 的性能瓶颈",
  "parts": [
    {
      "type": "text",
      "text": "请分析下面这段 SQL 的性能瓶颈"
    }
  ],
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

字段说明：

- `content`
  - 纯文本摘要字段，便于兼容和列表展示
- `parts`
  - 真正的结构化消息内容
  - 当前支持：
    - `text`
    - `image`
- `provider`
  - provider ID
- `model`
  - 模型 ID

如果省略 `provider/model`，后端会使用当前会话最近一次 assistant 使用的模型，或 provider 默认模型。

## 3. SSE 返回事件

聊天和重试接口都会返回 `text/event-stream`。

当前会用到这些事件：

- `message-start`
  - 返回创建好的用户消息和 assistant 消息
- `status`
  - `pending / streaming / completed / stopped / error`
- `text-delta`
  - 文本增量
- `tool-call`
  - 模型触发工具调用
- `tool-result`
  - 工具返回结果
- `finish`
  - 一轮生成结束
- `error`
  - 生成前置失败或运行时错误

## 4. 停止、重试、编辑、删除

### 4.1 停止当前生成

```text
POST /api/chat/conversations/:id/messages/:messageId/stop
```

用途：

- 主动停止一条仍在生成中的 assistant 消息

返回：

- 更新后的消息对象，`status = "stopped"`

### 4.2 重试最后一条 AI 回复

```text
POST /api/chat/conversations/:id/messages/:messageId/retry
```

说明：

- 只允许重试“当前对话最后一条 assistant 回复”
- 会原地重用这条 assistant 消息 ID
- 同样返回 SSE

可选请求体：

```json
{
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

### 4.3 修改消息

```text
PATCH /api/chat/conversations/:id/messages/:messageId
```

用途：

- 直接修改一条已有消息
- 不会自动重跑生成

### 4.4 删除消息

```text
DELETE /api/chat/conversations/:id/messages/:messageId
```

用途：

- 删除当前消息
- 不会级联删除后续消息

## 5. 图片输入与 Vision Fallback

### 5.1 图片请求格式

```json
{
  "content": "请描述这张图片",
  "parts": [
    {
      "type": "image",
      "image": "data:image/png;base64,...",
      "mimeType": "image/png"
    },
    {
      "type": "text",
      "text": "请描述这张图片"
    }
  ],
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

### 5.2 后端处理逻辑

当目标模型支持图片输入时：

- 保留原始图片 part

当目标模型不支持图片输入时：

1. 查当前会话图片转述缓存
2. 缓存命中则直接复用
3. 未命中则调用 `visionFallback`
4. 把转述文本替换成文本 part 继续送给主聊天模型

### 5.3 错误行为

如果图片转述失败：

- 后端不会吞错
- 会把原始错误通过 SSE `error` 事件直接返回给前端

## 7. 后端内部扩展时怎么调用模型

这一节说的是“后端服务内部新增业务代码”时，应该怎么复用现有统一调用链，而不是重新直连某家 SDK。

### 7.1 推荐分层

有两种常见场景：

- 你要复用“完整聊天语义”
  - 例如：落库用户消息、创建 assistant 占位消息、后台继续生成、SSE 转发、停止 / 重试 / 编辑 / 删除
  - 这种场景不要自己拼 `streamText`
  - 直接走 `ChatMessageService`
- 你只是在某个后端服务里做一次模型调用
  - 例如：写 SQL 性能分析、代码审查、自动摘要、图片说明
  - 这种场景走统一 AI 门面：
    - `AiProviderService`
    - `ChatMessageTransformService`
    - `sdk-adapter.ts` 里的 `runGenerateText` / `runStreamText`

### 7.2 统一调用链应该怎么走

推荐顺序：

1. 先用 `AiProviderService.getModelConfig(providerId, modelId)` 解析最终模型配置和能力
2. 如果输入里可能有图片，先把消息整理成聊天运行时消息，再交给 `ChatMessageTransformService.transformMessages(...)`
3. 用 `toAiSdkMessages(...)` 把运行时消息转成 AI SDK 消息
4. 最后通过 `runGenerateText(...)` 或 `runStreamText(...)` 调模型

这样做的原因是：

- provider 选型、默认模型和能力读取都统一走 `AiProviderService`
- 图片降级、Vision Fallback 和会话级图片转述缓存统一走 `ChatMessageTransformService`
- 复杂的 AI SDK 泛型隔离在 `sdk-adapter.ts`
- 业务模块不用自己 import `@ai-sdk/openai` / `@ai-sdk/anthropic` / `@ai-sdk/google`

### 7.3 服务内一次性调用示例

下面这个例子适合“某个后端服务里直接发起一次分析请求”：

```ts
import { Injectable } from '@nestjs/common';
import { AiProviderService } from '../ai/ai-provider.service';
import { runGenerateText } from '../ai/sdk-adapter';
import { ChatMessageTransformService } from '../chat/chat-message-transform.service';
import { toAiSdkMessages } from '../chat/sdk-message-converter';
import type { ChatRuntimeMessage } from '../chat/chat-message-session';

@Injectable()
export class SqlReviewService {
  constructor(
    private readonly aiProvider: AiProviderService,
    private readonly messageTransform: ChatMessageTransformService,
  ) {}

  /**
   * 分析 SQL 性能问题。
   * @param conversationId 逻辑会话 ID，用于图片降级缓存等上下文复用
   * @param providerId provider ID
   * @param modelId 模型 ID
   * @param sql 待分析 SQL
   * @returns 模型返回的文本结果
   */
  async reviewSql(
    conversationId: string,
    providerId: string,
    modelId: string,
    sql: string,
  ): Promise<string> {
    const modelConfig = this.aiProvider.getModelConfig(providerId, modelId);
    const runtimeMessages: ChatRuntimeMessage[] = [
      {
        role: 'user',
        content: [
          {
            type: 'text',
            text: `请分析下面 SQL 的性能瓶颈，并给出索引建议：\n${sql}`,
          },
        ],
      },
    ];
    const transformedMessages = await this.messageTransform.transformMessages(
      conversationId,
      runtimeMessages,
      modelConfig,
    );
    const result = await runGenerateText({
      model: this.aiProvider.getModel(providerId, modelId),
      messages: toAiSdkMessages(transformedMessages),
    });

    return result.text;
  }
}
```

### 7.4 什么时候用 `runGenerateText`，什么时候用 `runStreamText`

- `runGenerateText`
  - 后端只需要最终完整文本
  - 例如：批处理摘要、自动分析、离线任务
- `runStreamText`
  - 你需要增量结果
  - 例如：SSE、WebSocket、长任务实时回显

当前聊天主链路就是：

- `ChatMessageService` 负责准备消息、模型、工具和降级
- `ChatTaskService` 负责后台持续执行、写库和广播事件
- `ChatController` 只负责把后台事件转发成 SSE

### 7.5 图片输入和逻辑会话 ID

`ChatMessageTransformService.transformMessages(...)` 需要传 `conversationId`，因为图片转述缓存是按“会话 + 图片”复用的。

所以：

- 如果你是在真实聊天会话里调用，直接传聊天会话 ID
- 如果你是在别的后端模块里扩展能力，也要传一个稳定的逻辑上下文 ID
  - 例如：`task:<taskId>`
  - 或：`automation:<automationId>`

不要每次都传随机 ID，否则同一上下文里的相同图片没法命中缓存。

### 7.6 什么时候不要直接调统一 SDK 适配层

以下场景优先复用现有服务，不要自己手搓调用：

- 需要聊天消息落库、恢复、停止、重试
  - 直接走 `ChatMessageService`
- 需要 provider 诊断或“测试连接”
  - 直接走 `AiProviderDiagnosticsService`
- 需要 Vision Fallback 单独转述图片
  - 直接走 `ImageToTextService`

### 7.7 新增 provider 支持时改哪里

如果以后要扩展 provider 能力：

- 官方 provider 目录和默认信息：
  - `packages/server/src/ai/official-provider-catalog.ts`
  - 当前目录包含 `openai`、`anthropic`、`gemini`、`groq`、`xai`、`mistral`、`cohere`、`cerebras`、`deepinfra`、`togetherai`、`perplexity`、`gateway`、`vercel`、`openrouter`
- 运行时 provider 注册逻辑：
  - `packages/server/src/ai/ai-provider.service.ts`
  - `packages/server/src/ai/ai-provider.helpers.ts`
- 兼容 provider 请求格式：
  - 当前只允许 `openai` / `anthropic` / `gemini`
  - 不要再新增别的兼容格式分支

业务模块本身不应该直接依赖某一家 provider SDK。

## 8. 实战示例

### 8.1 让模型做性能分析

```json
{
  "content": "请分析下面这段 SQL 的性能瓶颈，并给出索引建议：SELECT * FROM orders WHERE user_id = 42 ORDER BY created_at DESC LIMIT 50",
  "parts": [
    {
      "type": "text",
      "text": "请分析下面这段 SQL 的性能瓶颈，并给出索引建议：SELECT * FROM orders WHERE user_id = 42 ORDER BY created_at DESC LIMIT 50"
    }
  ],
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

适合：

- SQL 性能分析
- 慢查询解释
- 索引建议
- 代码复杂度审查

### 8.2 让模型做代码评审

```json
{
  "content": "请从性能、可维护性和潜在并发问题三个角度审查下面这段 TypeScript 代码：...",
  "parts": [
    {
      "type": "text",
      "text": "请从性能、可维护性和潜在并发问题三个角度审查下面这段 TypeScript 代码：..."
    }
  ],
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

### 8.3 图片理解

```json
{
  "content": "请告诉我图里主要内容",
  "parts": [
    {
      "type": "image",
      "image": "data:image/png;base64,...",
      "mimeType": "image/png"
    },
    {
      "type": "text",
      "text": "请告诉我图里主要内容"
    }
  ],
  "provider": "ds2api",
  "model": "deepseek-reasoner"

```

如果聊天模型本身不支持图片，后端会自动走 Vision Fallback。

## 9. 当前已验证的后端接口

截至 2026-03-26，本仓库已实际烟测通过：

- `auth`
  - 注册 / 登录 / 刷新
- `users`
  - 当前用户 / 列表 / 查询 / 更新 / 改角色 / 删除
- `ai`
  - provider 目录 / provider CRUD / 模型 CRUD / 默认模型 / 能力更新 / 连接测试 / 模型拉取 / vision fallback 读写
- `chat`
  - 会话创建 / 查询 / 删除
  - 发送 / 带图发送 / 重试 / 停止 / 修改消息 / 删除消息
- `memories`
  - 列表 / 删除
- `plugins`
  - 列表 / 已连接插件 / 删除
- `automations`
  - 创建 / 列表 / 查询 / 切换 / 执行 / 日志 / 删除
