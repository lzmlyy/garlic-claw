# 后端模型调用接口说明

本文档说明 Garlic Claw 当前后端如何配置 provider、如何统一调用模型，以及聊天接口、后端内部扩展调用、停止 / 重试 / 图片链路。

## 1. provider 配置入口

### 1.1 配置文件

AI provider 不再从环境变量读取，统一放在：

```text
packages/server/tmp/ai-settings.server.json
```

如果设置了：

```text
GARLIC_CLAW_AI_SETTINGS_PATH
```

则会改读该环境变量指定的路径。

项目里保留的模板文件是：

```text
config/ai-settings.example.json
```

如果你希望把配置固定到仓库根目录，可先复制：

```bash
cp config/ai-settings.example.json config/ai-settings.json
```

再把环境变量指到这个文件。

当前示例文件已经包含：

- provider 列表
- `visionFallback`
- `hostModelRouting`

### 1.2 支持的 provider 类型

当前只支持两类 provider 接入方式：

- catalog provider
  - `mode: "catalog"`
  - `driver` 使用 catalog 项
- 协议接入
  - `mode: "protocol"`
  - `driver` 只允许：
    - `openai`
    - `anthropic`
    - `gemini`

### 1.3 协议族

当前真实运行映射如下：

- `openai` -> `@ai-sdk/openai`
- `anthropic` -> `@ai-sdk/anthropic`
- `gemini` -> `@ai-sdk/google`

协议接入会复用对应 driver 的协议族适配，其中 `openai` 协议接入固定走 `/v1/chat/completions`。

## 2. 统一聊天入口

后端统一对外暴露聊天接口：

```text
POST /api/chat/conversations/:id/messages
```

这个接口负责：

- 读取 `provider` / `model`
- 读取模型能力配置
- 处理图片输入与 Vision Fallback
- 创建用户消息和 assistant 占位消息
- 以 SSE 增量返回生成过程

### 2.1 请求体

```json
{
  "content": "请分析下面这段 SQL 的性能瓶颈",
  "parts": [
    {
      "type": "text",
      "text": "请分析下面这段 SQL 的性能瓶颈"
    }
  ],
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

字段说明：

- `content`
  - 纯文本摘要字段，便于兼容和列表展示
- `parts`
  - 真正的结构化消息内容
  - 当前支持：
    - `text`
    - `image`
- `provider`
  - provider ID
- `model`
  - 模型 ID

如果省略 `provider/model`，后端会使用当前会话最近一次 assistant 使用的模型，或 provider 默认模型。

## 3. SSE 返回事件

聊天和重试接口都会返回 `text/event-stream`。

当前会用到这些事件：

- `message-start`
  - 返回创建好的用户消息和 assistant 消息
- `status`
  - `pending / streaming / completed / stopped / error`
- `text-delta`
  - 文本增量
- `tool-call`
  - 模型触发工具调用
- `tool-result`
  - 工具返回结果
- `finish`
  - 一轮生成结束
- `error`
  - 生成前置失败或运行时错误

## 4. 停止、重试、编辑、删除

### 4.1 停止当前生成

```text
POST /api/chat/conversations/:id/messages/:messageId/stop
```

用途：

- 主动停止一条仍在生成中的 assistant 消息

返回：

- 更新后的消息对象，`status = "stopped"`

### 4.2 重试最后一条 AI 回复

```text
POST /api/chat/conversations/:id/messages/:messageId/retry
```

说明：

- 只允许重试“当前对话最后一条 assistant 回复”
- 会原地重用这条 assistant 消息 ID
- 同样返回 SSE

可选请求体：

```json
{
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

### 4.3 修改消息

```text
PATCH /api/chat/conversations/:id/messages/:messageId
```

用途：

- 直接修改一条已有消息
- 不会自动重跑生成

### 4.4 删除消息

```text
DELETE /api/chat/conversations/:id/messages/:messageId
```

用途：

- 删除当前消息
- 不会级联删除后续消息

## 5. 图片输入与 Vision Fallback

### 5.1 图片请求格式

```json
{
  "content": "请描述这张图片",
  "parts": [
    {
      "type": "image",
      "image": "data:image/png;base64,...",
      "mimeType": "image/png"
    },
    {
      "type": "text",
      "text": "请描述这张图片"
    }
  ],
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

### 5.2 后端处理逻辑

当目标模型支持图片输入时：

- 保留原始图片 part

当目标模型不支持图片输入时：

1. 查当前会话图片转述缓存
2. 缓存命中则直接复用
3. 未命中则调用 `visionFallback`
4. 把转述文本替换成文本 part 继续送给主聊天模型

### 5.3 错误行为

如果图片转述失败：

- 后端不会吞错
- 会把原始错误通过 SSE `error` 事件直接返回给前端

## 7. 后端内部扩展时怎么调用模型

这一节说的是“后端服务内部新增业务代码”时，应该怎么复用现有统一调用链，而不是重新直连某家 SDK。

### 7.1 推荐分层

有三种常见场景：

- 你要复用“完整聊天语义”
  - 例如：落库用户消息、创建 assistant 占位消息、后台继续生成、SSE 转发、停止 / 重试 / 编辑 / 删除
  - 这种场景不要自己拼 `streamText`
  - 当前 owner 是：
    - `ConversationMessageLifecycleService`
    - `ConversationTaskService`
    - `ConversationController`
- 你只是在某个后端服务里做一次模型调用
  - 例如：SQL 性能分析、代码审查、自动摘要
  - 直接走 `AiModelExecutionService`
- 你需要图片转述或图片输入降级
  - 直接走 `AiVisionService`

当前统一调用层已经收口到：

- `AiModelExecutionService`
  - provider 选择
  - 默认模型回退
  - `variant / headers / providerOptions / maxOutputTokens`
  - `generateText(...)`
  - `streamText(...)`
- `AiVisionService`
  - `conversationId + image` 级别图片转述缓存
  - `visionFallback`
  - `ChatMessagePart[]` 图片 part 转文本补充
- `ConversationMessageLifecycleService`
  - 对话消息落库
  - hook mutation
  - tool 集合装配
  - 聊天消息到模型消息的最终准备

### 7.2 统一调用链应该怎么走

推荐顺序：

1. 普通后端调用先注入 `AiModelExecutionService`
2. 如果输入里带图片或要复用图片转述缓存，再额外注入 `AiVisionService`
3. 把输入整理成 `PluginLlmMessage[]`
4. 调用：
   - `generateText(...)`
   - 或 `streamText(...)`
5. 如果要走完整聊天语义，则进入 `ConversationMessageLifecycleService`，不要绕开它自己拼聊天主链

这样做的原因是：

- provider 选型、默认模型和 fallback 统一走 `AiModelExecutionService`
- 图片转述、Vision Fallback 和会话级缓存统一走 `AiVisionService`
- 对话主链的 hook mutation、工具装配、消息持久化统一走 `ConversationMessageLifecycleService`
- 业务模块不需要自己串接旧的 chat / provider 中间层

### 7.3 服务内一次性调用示例

下面这个例子适合“某个后端服务里直接发起一次分析请求”：

```ts
import { Injectable } from '@nestjs/common';
import { AiModelExecutionService } from '../ai/ai-model-execution.service';
import type { PluginLlmMessage } from '@garlic-claw/shared';

@Injectable()
export class SqlReviewService {
  constructor(
    private readonly aiModelExecutionService: AiModelExecutionService,
  ) {}

  async reviewSql(
    providerId: string,
    modelId: string,
    sql: string,
  ): Promise<string> {
    const messages: PluginLlmMessage[] = [
      {
        role: 'user',
        content: [
          {
            type: 'text',
            text: `请分析下面 SQL 的性能瓶颈，并给出索引建议：\n${sql}`,
          },
        ],
      },
    ];

    const executed = await this.aiModelExecutionService.generateText({
      providerId,
      modelId,
      system: '你是一个 SQL 性能分析助手',
      messages,
      variant: 'reasoningHigh',
      providerOptions: {
        reasoning: {
          budget: 2048,
        },
      },
      headers: {
        'X-Request-Source': 'sql-review-service',
      },
      maxOutputTokens: 1024,
    });

    return executed.text;
  }
}
```

参数覆盖顺序是：

1. `model.options`
2. `model.variants[variant]`
3. 调用时显式传入的 `providerOptions`

请求头和输出 token 也是同样思路：

- `headers`: `model.headers -> 调用时 headers`
- `maxOutputTokens`: `显式传入 -> model.limit?.output`

### 7.4 什么时候用 `generateText`，什么时候用 `streamText`

- `AiModelExecutionService.generateText`
  - 适合一次性文本生成
  - 例如：provider 测试连接、后台分析任务、图片转述
- `AiModelExecutionService.streamText`
  - 适合业务服务自己消费流式结果
  - 例如：需要边生成边处理增量文本
- `ConversationMessageLifecycleService`
  - 适合完整聊天主链
  - 会在内部调用 `AiVisionService`、`ToolRegistryService`、`AiModelExecutionService`
- `ConversationTaskService`
  - 负责流式任务执行、状态写回和 SSE 事件广播

当前聊天主链路就是：

- `ConversationMessageLifecycleService` 负责准备消息、模型、工具和 hook mutation
- `ConversationTaskService` 负责后台持续执行、写库和广播事件
- `ConversationController` 负责把后台事件转发成 SSE

### 7.5 图片输入和逻辑会话 ID

`AiVisionService.resolveMessageParts(...)` 需要传 `conversationId`，因为图片转述缓存是按“会话 + 图片”复用的。

所以：

- 如果你是在真实聊天会话里调用，直接传聊天会话 ID
- 如果你是在别的后端模块里扩展能力，也要传一个稳定的逻辑上下文 ID
  - 例如：`task:<taskId>`
  - 或：`automation:<automationId>`

不要每次都传随机 ID，否则同一上下文里的相同图片没法命中缓存。

### 7.6 什么时候不要直接调统一 SDK 适配层

以下场景优先复用现有服务，不要自己手搓调用：

- 需要聊天消息落库、恢复、停止、重试
  - 直接走 `ConversationMessageLifecycleService`
- 需要 provider 诊断或“测试连接”
  - 直接走现有 AI 管理接口对应的 service
- 需要 Vision Fallback 单独转述图片
  - 直接走 `AiVisionService`

### 7.7 新增 provider 支持时改哪里

如果以后要扩展 provider 能力：

- catalog provider 目录和默认信息：
  - `packages/server/src/ai-management/ai-provider-catalog.ts`
  - 当前目录包含：
    - `openai`
    - `anthropic`
    - `gemini`
- 运行时 provider 注册逻辑：
  - `packages/server/src/ai-management/ai-management.service.ts`
  - `packages/server/src/ai-management/ai-provider-settings.service.ts`
  - 当前运行时只保留：
    - `@ai-sdk/openai`
    - `@ai-sdk/anthropic`
    - `@ai-sdk/google`
- 协议接入协议族：
  - 当前只允许 `openai` / `anthropic` / `gemini`
  - 不要再新增别的协议族分支

业务模块本身不应该直接依赖某一家 provider SDK。

## 8. 实战示例

### 8.1 让模型做性能分析

```json
{
  "content": "请分析下面这段 SQL 的性能瓶颈，并给出索引建议：SELECT * FROM orders WHERE user_id = 42 ORDER BY created_at DESC LIMIT 50",
  "parts": [
    {
      "type": "text",
      "text": "请分析下面这段 SQL 的性能瓶颈，并给出索引建议：SELECT * FROM orders WHERE user_id = 42 ORDER BY created_at DESC LIMIT 50"
    }
  ],
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

适合：

- SQL 性能分析
- 慢查询解释
- 索引建议
- 代码复杂度审查

### 8.2 让模型做代码评审

```json
{
  "content": "请从性能、可维护性和潜在并发问题三个角度审查下面这段 TypeScript 代码：...",
  "parts": [
    {
      "type": "text",
      "text": "请从性能、可维护性和潜在并发问题三个角度审查下面这段 TypeScript 代码：..."
    }
  ],
  "provider": "ds2api",
  "model": "deepseek-reasoner"
}
```

### 8.3 图片理解

```json
{
  "content": "请告诉我图里主要内容",
  "parts": [
    {
      "type": "image",
      "image": "data:image/png;base64,...",
      "mimeType": "image/png"
    },
    {
      "type": "text",
      "text": "请告诉我图里主要内容"

```

如果聊天模型本身不支持图片，后端会自动走 Vision Fallback。

  - 列表 / 删除
- `plugins`
  - 列表 / 已连接插件 / 健康 / 配置 / scopes / 事件 / actions / route 转发 / 删除

  - 创建 / 列表 / 查询 / 切换 / 执行 / 日志 / 删除
