# 扩展内核维护说明

本文档面向维护者，说明 Garlic Claw 当前 extension kernel 的收口方向，以及 `plugin host core` / `plugin runtime core` 继续减法时的边界判断。

## 目标

当前项目的长期方向不是继续把能力堆进本体，而是把本体收敛成稳定内核：

- core 只保留宿主边界、治理和最少运行时原语
- 作者侧继续兼容 `plugin / MCP / skill`
- 统一发生在 runtime contract，不强制统一 authoring format

如果需要看“当前 contract 到底是什么、哪些原语已经正式固化”，先读：

- [`docs/扩展内核契约说明.md`](./扩展内核契约说明.md)

这意味着维护时应优先做两件事：

1. 把可证明是纯逻辑、纯投影、纯校验的内容移出大 service
2. 避免为了“方便”重新引入 core 特判和裸对象直连

## 当前分层

### PluginHostService

`PluginHostService` 继续保留：

- Host API 分发表
- 需要调用 Nest service / Prisma / AI execution service 的宿主行为
- 会话与用户权限校验

已迁出的 helper 主要承担：

- 参数读取与基础 JSON reader
- 当前 provider 摘要投影
- provider summary 查找 / not-found 归一化
- provider model 摘要投影
- LLM generate 参数与结果投影
- conversation / memory / user / message 摘要序列化

判断原则：

- 如果逻辑只依赖 `context + params + plain object`，应优先落到 helper
- 如果逻辑要访问 Prisma、宿主 service 或需要抛宿主语义异常，保留在 HostService

### PluginRuntimeService

`PluginRuntimeService` 继续保留：

- builtin / remote transport 统一调度
- Host API 权限校验
- hook 执行编排
- execution slot 并发保护
- 运行时 record map 与治理快照更新

已迁出的 helper 主要分为几类：

- `plugin-runtime-input.helpers.ts`
  - runtime host param / timeout / subagent request 读取
- `plugin-runtime-clone.helpers.ts`
  - hook payload clone / assistant output normalize
- `plugin-runtime-validation.helpers.ts`
  - 基础结构校验与 path normalize
- `plugin-runtime-hook-*.helpers.ts`
  - hook filter / result normalize / mutation apply
- `plugin-runtime-dispatch.helpers.ts`
  - runtime record 查找
  - scope 启用断言
  - hook dispatch 记录筛选与稳定排序
- `plugin-runtime-session.helpers.ts`
  - conversation session map 状态机
  - runtime-facing start/get/keep/finish/list/info bridge
- `plugin-runtime-record.helpers.ts`
  - 并发上限、pressure snapshot、supported actions
- `plugin-runtime-subagent.helpers.ts`
  - subagent request/result/tool-set request 组装
  - resolved short-circuit result 投影
- `plugin-runtime-manifest.helpers.ts`
  - lifecycle/self manifest 投影与 route/hook/tool 查找
- `plugin-runtime-timeout.helpers.ts`
  - Promise timeout 包装

判断原则：

- 如果逻辑不依赖 `this.records / transport / ModuleRef / pluginService`，优先外提
- 如果逻辑只是在做“查 manifest / 投影结果 / 包 Promise / 组装 request/result”，不要继续留在 service
- 如果逻辑承担真正的 runtime 时序控制、错误记录、治理更新，保留在 RuntimeService

## 后续减法优先级

当前仍值得继续处理的方向：

1. `PluginHostService`
   - 继续外提 provider 相关的查找与投影薄壳
   - 继续审视 `generateCore(...)` 周围是否还有纯组装逻辑
2. `PluginRuntimeService`
   - 继续压缩只剩单点包装的 helper bridge
   - 审视 execution slot / failure dispatch 周围是否还能形成更清晰的 helper 边界
3. 文档同步
   - 每次新增 helper 边界后，更新 `TODO.md`
   - 如果边界判断规则发生变化，同时更新本文档

## 不该做的事

- 不要因为局部方便，把 helper 再塞回大 service
- 不要新增只服务某一个 builtin 的 core 特判路径
- 不要让插件、MCP、skill 之间通过裸对象直接互调
- 不要把“临时兼容”变成长期默认路径

## 验证要求

任何这类减法改动完成后，至少要补：

- 定向 helper spec
- 相关 service spec
- `npm run typecheck -w packages/server`
- `cd packages/server && npm test -- --runInBand`
- `cd packages/server && npm run build`
- `npm run lint`
