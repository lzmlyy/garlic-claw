# 扩展内核维护说明

本文档面向维护者，说明 Garlic Claw 当前 extension kernel 的真实 owner 边界，以及继续做减法时应该优先守住什么。

## 目标

当前长期方向不是继续把能力堆回本体，而是把后端收敛成稳定内核：

- core 只保留宿主边界、治理和最少运行时原语
- 作者侧继续兼容 `plugin / MCP / skill`
- 不再回到“一个超厚 runtime service 包办一切”的旧结构

如果需要看当前 contract 本身，先读：

- [`docs/扩展内核契约说明.md`](./扩展内核契约说明.md)

## 当前 owner

### `RuntimeHostService`

只保留需要宿主协作的 Host API 分发：

- 对外暴露给插件的 host call 入口
- 需要协同 Nest service / AI execution / user service 的宿主行为
- 需要组合多个明确 owner 的宿主级流程

不应再把纯投影、纯 reader、纯校验重新塞回这里。

### `RuntimeHostPluginDispatchService`

负责统一插件 dispatch 语义：

- `tool / hook / route` 的 builtin / remote 调度
- runtime plugin snapshot 列表
- 面向 `tool`、`conversation`、`automation`、HTTP route 的统一插件调用

如果逻辑主要是在做插件选择、hook 过滤、remote request 往返、builtin handler 调度，应优先留在这里或其直接协作 owner，而不是绕回 `RuntimeHostService`。

### `RuntimeHostPluginRuntimeService`

负责插件自己的 runtime 资源面：

- `storage.*`
- `state.*`
- `config.*`
- `cron.*`

这类插件私有状态与资源管理不应再混回 dispatch owner 或治理 owner。

### `RuntimeHostKnowledgeService` / `RuntimeHostUserContextService`

负责宿主本地读取视图：

- provider / persona / kb / memory / user 等读取侧语义
- 当前用户或上下文相关的最小宿主摘要

如果逻辑只是读取或投影宿主现有数据，应优先放在这些明确 reader owner。

### `RuntimeHostSubagentRunnerService`

负责子代理任务执行与写回：

- request 组装
- tool set 选择
- 运行结果写回与任务视图

子代理能力不要重新塞回通用 runtime owner。

### `RuntimePluginGovernanceService`

现在只保留治理 owner：

- `health-check`
- `reload`
- `reconnect`
- `supportedActions`

它不再承担通用 dispatch，也不再保留薄转发壳。

### `PluginPersistenceService`

负责插件持久化与 append-only 事件日志：

- config / scopes
- plugin event log
- 与插件持久化记录相关的写入与查询

文档、控制器和 gate 都要按这个 owner 名维护，不能再写回旧壳。

## 维护规则

### 1. 不回退到旧壳

不再新增或恢复这些做法：

- 泛化 `helper / facade / compatibility` 壳
- “只是换名字”的薄转发 service
- 旧式 `plugin runtime` catch-all owner

### 2. 只保留明确 owner

遇到以下逻辑时，直接并回明确边界：

- 纯 dispatch：`RuntimeHostPluginDispatchService`
- 纯 runtime 资源：`RuntimeHostPluginRuntimeService`
- 纯治理：`RuntimePluginGovernanceService`
- 纯持久化 / 事件日志：`PluginPersistenceService`
- 纯读取视图：对应 reader owner

### 3. 文档与 gate 同步

每次删旧路径或 owner 时，同批必须同步：

- `TODO.md`
- `docs/**`
- `tools/refactor/config/zero-legacy-refs.config.json`
- 其他相关 gate / boundary map

不能让旧 owner 只是在代码里消失，文档和门禁还继续放行。

## 验证要求

这类改动完成后，至少要补：

- 对应 owner 的定向 spec
- `cd packages/server && npx tsc --noEmit`
- `cd packages/server && npm test -- --runInBand`
- `cd packages/server && npm run build`
- `cd packages/server && npm run smoke:http`
- `npm run repo:zero-legacy-refs -- --scope server-cutover`
- `npm run repo:refactor-metrics`
- `git diff --check`
