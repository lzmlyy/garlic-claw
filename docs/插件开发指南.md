# 插件开发指南

Garlic Claw 现在使用统一插件运行时：

- `builtin`：内建插件，跟随后端启动，但仍走同一套 manifest、Hook、Host API 和执行协议
- `remote`：远程插件，通过 WebSocket 接入，同样走这套协议

当前不做插件市场，重点是统一协议和统一运行时。

这份文档面向插件作者，重点说明“怎么声明插件、怎么接 Hook、怎么调用 Host API、怎么调试和治理插件”。面向内部架构演进的状态判断、AstrBot 对照和后续路线，统一记录在项目计划文件中，不放在这里。维护者如果需要看 extension kernel 当前的 helper 分层和减法边界，请同时参考：

- [`docs/扩展内核契约说明.md`](./扩展内核契约说明.md)
- [`docs/扩展内核维护说明.md`](./扩展内核维护说明.md)

## 概述

插件系统统一了这些概念：

- `manifest`：插件元数据、工具列表、Hook 列表、权限
- `crons`：插件声明或动态注册的定时任务
- `config schema`：插件可声明自己的配置项、默认值和字段类型
- `tools`：可被 AI 或自动化调用的工具
- `hooks`：宿主在关键阶段触发的扩展点
- `routes`：插件可向宿主声明的 HTTP JSON Route
- `host api`：插件访问宿主能力的统一入口
- `transport`：只有传输方式不同，`builtin` 走进程内，`remote` 走 WebSocket

## 远程插件快速开始

### 1. 启动服务

建议先启动开发环境：

Windows：

```powershell
.\start.bat restart
```

Linux / WSL：

```bash
bash ./start.sh restart
```

直接调用 Python 主入口也可以：

```bash
python tools/一键启停脚本.py restart
```

如果你想前台直接看后端和前端日志，再改用：

```bash
python tools/一键启停脚本.py --tail-logs
```

默认插件 WebSocket 地址：

```text
ws://127.0.0.1:23331
```

### 2. 创建插件项目

```bash
mkdir packages/plugins/plugin-my-device
cd packages/plugins/plugin-my-device
npm init -y
```

`package.json`：

```json
{
  "name": "@garlic-claw/plugin-my-device",
  "version": "0.1.0",
  "private": true,
  "main": "dist/index.js",
  "scripts": {
    "build": "tsc",
    "start": "node dist/index.js"
  },
  "dependencies": {
    "@garlic-claw/plugin-sdk": "*",
    "@garlic-claw/shared": "*"
  },
  "devDependencies": {
    "typescript": "^5.7.0"
  }
}
```

### 3. 编写插件代码

```typescript
import { DeviceType, type PluginCapability } from '@garlic-claw/shared';
import { PluginClient } from '@garlic-claw/plugin-sdk/client';

const tools: PluginCapability[] = [
  {
    name: 'get_status',
    description: '获取设备状态',
    parameters: {},
  },
  {
    name: 'set_brightness',
    description: '设置亮度',
    parameters: {
      level: {
        type: 'number',
        description: '亮度 0-100',
        required: true,
      },
    },
  },
];

const client = PluginClient.fromBootstrap({
  serverUrl: process.env.WS_URL ?? 'ws://127.0.0.1:23331',
  token: process.env.PLUGIN_TOKEN ?? '',
  pluginName: 'my-device-living-room',
  deviceType: DeviceType.IOT,
  tokenExpiresIn: process.env.PLUGIN_TOKEN_EXPIRES_IN ?? '30d',
}, {
  manifest: {
    name: 'Living Room Device',
    version: '1.0.0',
    description: '客厅设备插件',
    permissions: [],
    tools,
    hooks: [],
    config: {
      fields: [
        {
          key: 'displayName',
          type: 'string',
          description: '设备在回复中的显示名称',
          defaultValue: '客厅设备',
        },
      ],
    },
  },
});

client.onCommand('get_status', async () => {
  return {
    online: true,
    brightness: 75,
  };
});

client.onCommand('set_brightness', async (params) => {
  const level = params.level as number;
  return {
    success: true,
    brightness: level,
  };
});

client.connect();
```

如果你要直接使用 SDK 的不同能力面，当前推荐按入口分开引用：

- `@garlic-claw/plugin-sdk/client`
  - `PluginClient`
- `@garlic-claw/plugin-sdk/host`
  - host facade 类型与 helper
- `@garlic-claw/plugin-sdk/authoring`
  - authoring helper 与 builtin authoring utilities

根入口 `@garlic-claw/plugin-sdk` 仍保留兼容导出，但新插件代码优先使用这些子路径入口。

### 4. 在线生成接入信息并配置环境变量

先用管理员 access token 在线生成这一个远程插件实例的专用接入信息：

```bash
curl -X POST http://127.0.0.1:23330/api/plugins/remote/bootstrap \
  -H "Authorization: Bearer <admin-access-token>" \
  -H "Content-Type: application/json" \
  -d '{
    "pluginName": "my-device-living-room",
    "deviceType": "iot",
    "displayName": "Living Room Device",
    "description": "客厅设备插件",
    "version": "1.0.0"
  }'
```

返回值示例：

```json
{
  "pluginName": "my-device-living-room",
  "deviceType": "iot",
  "serverUrl": "ws://127.0.0.1:23331",
  "token": "<remote-plugin-token>",
  "tokenExpiresIn": "30d"
}
```

然后把返回的 `token` 和 `serverUrl` 写入环境变量：

```bash
export PLUGIN_TOKEN="<remote-plugin-token>"
export WS_URL="ws://127.0.0.1:23331"
```

这组返回值里的：

- `serverUrl`
- `token`
- `pluginName`
- `deviceType`

可以直接作为 `PluginClient` 的连接基础参数使用。

如果你保留了完整 bootstrap JSON，也可以直接：

```typescript
const client = PluginClient.fromBootstrap(bootstrap, {
  manifest: {
    name: 'Living Room Device',
    version: '1.0.0',
    permissions: [],
    tools,
  },
});
```

注意：

- 现在推荐始终使用 `POST /api/plugins/remote/bootstrap` 返回的专用远程插件令牌
- 网关为了兼容旧客户端，暂时仍接受管理员 JWT；但新接入流程不再建议这样做
- 如需对外暴露不同地址，可用 `REMOTE_PLUGIN_WS_URL` 覆盖返回的 `serverUrl`
- 可用 `REMOTE_PLUGIN_TOKEN_EXPIRES_IN` 调整专用令牌有效期

### 5. 构建并运行

```bash
npm run build
npm start
```

## Manifest

远程插件最终注册到服务器的 manifest 结构如下：

```typescript
interface PluginManifest {
  id: string;
  name: string;
  version: string;
  runtime: 'builtin' | 'remote';
  description?: string;
  permissions: PluginPermission[];
  tools: PluginCapability[];
  crons?: PluginCronDescriptor[];
  hooks?: PluginHookDescriptor[];
  config?: PluginConfigSchema;
  routes?: PluginRouteDescriptor[];
}
```

`hooks` 中的单项描述目前支持：

```typescript
interface PluginHookDescriptor {
  name: PluginHookName;
  description?: string;
  priority?: number;
  filter?: {
    message?: {
      commands?: string[];
      regex?: string | { pattern: string; flags?: string };
      messageKinds?: Array<'text' | 'image' | 'mixed'>;
    };
  };
}
```

补充说明：

- `priority` 数字越小越早执行；同优先级下宿主继续按插件 ID 稳定排序
- 当前 `filter` 只对 `message:received` 生效
- `commands / regex / messageKinds` 属于最小声明式过滤；更复杂条件仍应在 Hook 代码里自行判断

其中：

- `id` 由 `pluginName` 决定
- `runtime` 对远程插件固定为 `remote`
- `tools` 是 AI / 自动化可调用的工具
- `hooks` 是宿主扩展点
- `config` 用来声明插件配置项；配置值由宿主持久化，插件通过 Host API 读取
- `routes` 用来声明插件可处理的宿主 HTTP JSON Route

如果插件希望调低自己的运行时并发上限，可以在 `config` 中声明 `maxConcurrentExecutions` 这类数字字段；宿主当前会读取已解析配置中的同名值，并把它作为该插件 `tool / route / hook` 的统一并发上限。未声明时，宿主默认上限为 `6`。

## 工具处理器

`onCommand(name, handler)` 用来注册**工具处理器**，不是消息命令 DSL：

```typescript
client.onCommand('set_brightness', async (params, context) => {
  return {
    success: true,
    brightness: params.level,
  };
});
```

处理器签名：

```typescript
(params: JsonObject, context: PluginExecutionContext) => Promise<JsonValue> | JsonValue
```

`context` 中包含：

- `callContext`：当前是谁在调用这个工具，例如聊天工具调用、自动化调用
- `host`：统一 Host API 门面

## Hook 处理器

目前已内置的 Hook 包括：

- `message:received`
- `chat:before-model`
- `chat:waiting-model`
- `chat:after-model`
- `conversation:created`
- `message:created`
- `message:updated`
- `message:deleted`
- `automation:before-run`
- `automation:after-run`
- `tool:before-call`
- `tool:after-call`
- `response:before-send`
- `response:after-send`
- `plugin:loaded`
- `plugin:unloaded`
- `plugin:error`
- `cron:tick`

注册方式：

```typescript
client.onHook('chat:before-model', async (payload, context) => {
  return {
    action: 'mutate',
    systemPrompt: `${String((payload as {
      request?: { systemPrompt?: string }
    }).request?.systemPrompt ?? '')}\n\n这里可以追加系统提示词`,
  };
});
```

这个 Hook 适合做：

- 对话前上下文增强
- 记忆注入
- 会话级行为修正
- provider/model 路由
- 工具集合裁剪
- 在命中规则时直接 short-circuit

`message:received` 适合做：

- 在消息刚进入宿主时先监听、过滤、分流
- 命中命令或正则时改写待入库草稿与待送模型消息
- 在更前置的阶段直接 short-circuit

如果你想写 AstrBot 风格的“消息命令 / 命令组 / 别名”，现在更推荐优先使用 `plugin-sdk` 自带的作者体验层，而不是手写裸 `message:received` 分发：

- `client.onMessage(handler, { priority, filter })`
- `client.command(name, handler, { alias, priority, description })`
- `client.commandGroup(name, { alias, priority, description })`
- `client.sessionWaiter(handler)`

这些 helper 的语义：

- 都是建立在统一的 `message:received` Hook 之上，不会引入第二套私有协议
- SDK 会自动帮你合成 `message:received` 的 manifest descriptor
- `commandGroup(...)` 支持继续 `.command(...)` / `.group(...)`
- `sessionWaiter(...)` 会把“活动等待态命中的后续消息”优先交给本地 waiter handler
- group 被精确命中时，会自动生成一份子命令帮助文本
- handler 返回：
  - `string`
  - `{ content, parts? }`
  - 原生 `message:received` Hook 返回对象
  都会被 SDK 归一化处理
- 如果同时注册了裸 `client.onHook('message:received', ...)`，它会作为 DSL 管线后的兜底处理器

示例：

```typescript
const client = new PluginClient({
  serverUrl: process.env.WS_URL ?? 'ws://127.0.0.1:23331',
  token: process.env.PLUGIN_TOKEN!,
  pluginName: 'plugin.math-helper',
  deviceType: DeviceType.API,
});

client.command('help', () => '可用命令：/math add 1 2');

const math = client.commandGroup('math', {
  alias: ['m'],
  description: '数学命令',
});

math.command('add', ({ args }) => {
  const [left, right] = args.map((value) => Number(value));
  return `结果：${left + right}`;
}, {
  alias: ['sum'],
  description: '加法',
});

const calc = math.group('calc', {
  alias: ['c'],
  description: '高级计算',
});

calc.command('help', ({ matchedCommand, canonicalCommand }) => ({
  content: `你命中了 ${matchedCommand}（规范路径：${canonicalCommand}）`,
}));

client.onMessage((payload) => {
  if (payload.message.content === '你好') {
    return {
      action: 'short-circuit',
      assistantContent: '你好，我在。',
    };
  }

  return {
    action: 'pass',
  };
}, {
  filter: {
    regex: '^(你好|hello)$',
  },
});
```

`chat:waiting-model` 适合做：

- 记录“即将真正进入模型调用”的等待态
- 写入插件私有 KV / 事件日志
- 做轻量观测而不是改写请求

`chat:after-model` 适合做：

- 自动生成会话标题
- 回复完成后的摘要/提取
- 按回复结果触发轻量副作用
- 对最终 assistant 回复做收尾润色或统一改写
- 在发送前补齐或重写结构化 `parts`

`conversation:created` 适合做：

- 新会话初始化
- 会话级插件状态预热
- 记录会话级审计事件

`message:created` 适合做：

- 改写用户消息草稿
- 在入库前补充结构化 part
- 提前调整送给模型的消息快照

`message:updated` / `message:deleted` 适合做：

- 审计消息编辑与删除
- 维护插件私有索引或缓存
- 对编辑后的消息内容做受控修正

`automation:before-run` 适合做：

- 统一裁剪或改写自动化动作列表
- 命中规则时直接 short-circuit
- 在执行前写入插件私有状态

`automation:after-run` 适合做：

- 记录自动化执行摘要
- 写入插件 KV / 事件日志
- 对最终结果载荷做轻量改写

`tool:before-call` / `tool:after-call` 适合做：

- 工具调用审计
- 参数裁剪与补缺省值
- 对成功返回的工具结果做统一收尾

`response:before-send` / `response:after-send` 适合做：

- 最终回复发送前修饰
- 对最终回复的结构化 `parts` 做统一包装
- 发送后摘要与统计
- 把已发送结果写入插件私有 KV / 事件日志

`plugin:loaded` / `plugin:unloaded` / `plugin:error` 适合做：

- 插件治理审计
- 插件生命周期摘要
- 对失败插件做统一记录或告警转发

`cron:tick` 适合做：

- 周期巡检
- 定时同步
- 缓存清理
- 插件自身心跳或摘要任务

`chat:before-model` 当前输入是：

```typescript
{
  context: PluginCallContext,
  request: {
    providerId: string,
    modelId: string,
    systemPrompt: string,
    messages: PluginLlmMessage[],
    availableTools: PluginAvailableToolSummary[],
    variant?: string,
    providerOptions?: JsonObject,
    headers?: Record<string, string>,
    maxOutputTokens?: number,
  }
}
```

返回值支持三种：

- `action: 'pass'`
- `action: 'mutate'`
- `action: 'short-circuit'`

其中 `mutate` 可以改写：

- `providerId`
- `modelId`
- `systemPrompt`
- `messages`
- `toolNames`
- `variant`
- `providerOptions`
- `headers`
- `maxOutputTokens`

兼容说明：

- 旧版 `appendSystemPrompt` 返回结构仍可继续运行
- 推荐新插件直接使用强版 `pass / mutate / short-circuit`

`message:received` 当前输入是：

```typescript
{
  context: PluginCallContext,
  conversationId: string,
  providerId: string,
  modelId: string,
  message: PluginMessageHookInfo,
  modelMessages: PluginLlmMessage[],
}
```

返回值支持三种：

- `action: 'pass'`
- `action: 'mutate'`
- `action: 'short-circuit'`

其中 `mutate` 当前可改写：

- `providerId`
- `modelId`
- `content`
- `parts`
- `modelMessages`

`short-circuit` 会直接结束本轮模型调用，但用户消息仍会按当前载荷落库。

`chat:waiting-model` 当前输入是：

```typescript
{
  context: PluginCallContext,
  conversationId: string,
  assistantMessageId: string,
  providerId: string,
  modelId: string,
  request: ChatBeforeModelRequest,
}
```

这是观察型 Hook，当前不接收返回值，也不承担改写职责。

`chat:after-model` 当前输入是：

```typescript
{
  providerId: string,
  modelId: string,
  assistantMessageId: string,
  assistantContent: string,
  assistantParts: ChatMessagePart[],
  toolCalls: Array<{
    toolCallId: string,
    toolName: string,
    input: JsonValue,
  }>,
  toolResults: Array<{
    toolCallId: string,
    toolName: string,
    output: JsonValue,
  }>,
}
```

返回值支持两种：

- `action: 'pass'`
- `action: 'mutate'`

其中 `mutate` 当前最小支持改写：

- `assistantContent`
- `assistantParts`

当 `chat:after-model` 改写了 `assistantContent` 或 `assistantParts` 时：

- runtime 会统一归一化文本与 `parts`，保证旧插件只改文本时仍可兼容
- short-circuit 路径会二次回写数据库中的 assistant 消息
- 流式路径会在完成后再次持久化最终内容与 `parts`
- 前端会额外收到一条 `message-patch` SSE，用来覆盖最终回复文本 / `parts`

```typescript
    createdAt: string,

}
```

- `content`

- `modelMessages`
- `provider`
- `model`

插件现在还可以通过统一协议声明宿主内的 HTTP JSON Route。

```typescript

  deviceType: DeviceType.API,

    name: 'My HTTP Plugin',

    tools: [],
    hooks: [],
    routes: [
      {
        path: 'inspect/context',
        methods: ['GET'],
        description: '返回当前用户和会话上下文',
      },
    ],
  },
});

client.onRoute('inspect/context', async (request, context) => {
  const plugin = await context.host.getPluginSelf();
  const user = await context.host.getUser();

  return {
    status: 200,
    body: {
      plugin,
      user,
      query: request.query,

```

Route 处理器签名：

```typescript
(request: PluginRouteRequest, context: PluginExecutionContext) =>
  Promise<PluginRouteResponse> | PluginRouteResponse
```

当前 Route 入口路径统一为：

```text

```

例如：


这一层目前是：

- JSON Route
- 复用统一插件权限 / Host API / 超时与失败记录
- 由宿主认证用户后再转给插件处理

更推荐的作者体验层是：

- `context.host.conversationSession`
  - 这是对现有 `conversation.session.*` 的本地控制器封装
  - 可直接读取：

    - `session`
    - `timeoutMs`

  - 可直接调用：
    - `start(...)`
    - `get()`
    - `sync()`
    - `keep(...)`
    - `finish()`
- `client.sessionWaiter(handler)`
  - 用来把当前插件的某个本地 handler 绑定到活动会话等待态

  - 命中活动等待态时，waiter 会先于常规 `onMessage / command / commandGroup` 管线执行

典型写法：

```typescript
const profileWaiter = client.sessionWaiter(async (controller, payload) => {
  const text = payload.message.content?.trim() ?? '';

  if (!text) {
    return {
      action: 'short-circuit',
      assistantContent: '请直接发送文本内容。',
    };
  }

  if (text === '退出') {
    await controller.finish();
    return {
      action: 'short-circuit',
      assistantContent: '已退出当前多轮会话。',
    };
  }

  await controller.keep({
    timeoutMs: 60_000,
    resetTimeout: true,
  });

  return {
    action: 'short-circuit',
    assistantContent: `已记录你的名字：${text}`,
  };
});

client.command('profile', async (_input, context) => {
  await profileWaiter.start(context, {
    timeoutMs: 60_000,
    captureHistory: true,
    metadata: {
      step: 'name',
    },
  });

  return '请发送你的名字，输入“退出”可取消。';
});
```

如果你需要更底层的控制，原始 `startConversationSession / getConversationSession / keepConversationSession / finishConversationSession` 仍然保留。

`message.target.current.get` 用来读取当前执行上下文里的消息目标摘要：

- 当前单用户宿主里，返回值会映射为当前 `conversation` 目标
- 没有活动消息目标上下文时返回 `null`
- 需要 `conversation:read` 权限

`message.send` 用来让插件主动向当前或指定消息目标追加一条已完成的 assistant 消息：

- 默认写入当前执行上下文里的消息目标
- 也可以显式传 `target`，用于 cron / automation / subagent 这类延迟回写场景
- 当前第一版只支持 `target.type = 'conversation'`
- 当前第一版固定创建 `assistant` 消息，并复用 `message:created` Hook 变异链
- 需要 `conversation:write` 权限

`conversation.session.*` 用来让插件把当前会话切入“等待态”，从而优先接管后续用户输入：

- `startConversationSession({ timeoutMs, captureHistory, metadata })`
  - 启动当前会话的等待态
  - 后续用户消息会先进入当前插件的 `message:received`
- `getConversationSession()`
  - 读取当前插件在当前会话上的活动等待态
- `keepConversationSession({ timeoutMs, resetTimeout })`
  - 续期当前等待态
- `finishConversationSession()`
  - 结束当前等待态

第一版的 Garlic Claw 会话控制语义：

- 先收敛为“会话级”控制，不照搬 AstrBot 的平台 `session_filter`
- 活动等待态命中时，会先把消息交给归属插件处理，再决定是否继续进入 LLM
- `captureHistory: true` 时，等待态会累计命中的用户消息快照
- 当前也继续走 `conversation:write` 权限

当当前消息命中了活动等待态时，`message:received` payload 里会额外带上 `session`：

```typescript
client.onHook('message:received', async (payload, context) => {
  const hookPayload = payload as {
    session?: {
      conversationId: string,
      pluginId: string,
      timeoutMs: number,
      historyMessages: Array<{ role: string; content: string | null }>,
    } | null,
    message: {
      content: string | null,
    },
  };

  if (!hookPayload.session) {
    return {
      action: 'pass',
    };
  }

  if (hookPayload.message.content === '退出') {
    await context.host.finishConversationSession();
    await context.host.sendMessage({
      content: '已退出当前多轮会话。',
    });
    return {
      action: 'short-circuit',
      assistantContent: '已退出当前多轮会话。',
    };
  }

  await context.host.keepConversationSession({
    timeoutMs: 60_000,
    resetTimeout: true,
  });

  return {
    action: 'pass',
  };
});
```

- Cron 列表与 host cron 删除
- 治理动作

其中事件日志接口当前支持这些查询参数：

- `limit`
- `level`
- `type`
- `keyword`
- `cursor`

也就是说管理员现在不仅能看“最近一批事件”，还可以继续按服务端过滤与分页拉取日志，而不是只能在前端本地筛选一小段结果。

健康接口和插件列表里的 `health` 现在还会带上可选的 `runtimePressure`：

- `activeExecutions`
- `maxConcurrentExecutions`

宿主会对单插件的 `tool / route / hook` 执行统一做并发保护；当某个插件当前执行数已经打满时：

- 当前调用会被拒绝
- 事件日志会新增 `tool:overloaded` / `route:overloaded` / `hook:overloaded`
- 管理页可直接看到当前并发占用

插件列表接口现在还会为每个插件返回 `supportedActions`，前端和管理端应该以它作为治理按钮的事实来源，而不是再按 `runtimeKind` 猜测哪些动作可用。

当前已开放的治理动作：

- `health-check`
- `reload`
- `reconnect`

注意：

- 目前仍**不做插件市场 / 包安装 / 软件升级**
- 治理动作现在统一经由 `PluginController -> RuntimePluginGovernanceService` 分发；remote 动作再由 runtime gateway 执行，而不是控制器层直连 builtin / remote 实现
- `supportedActions` 由当前运行时 transport 显式声明；对于暂未接入 runtime 的持久化离线记录，宿主会保守回退为只暴露 `health-check`
- `reload` 对 `builtin` 表示重新装载定义
- `reload` 对当前在线 `remote` 插件表示主动断开当前连接，让其重新注册
- `reconnect` 对 `remote` 表示主动断开当前连接，交给插件 SDK 自动重连
- `DELETE /api/plugins/:name/crons/:jobId` 只允许删除 `source=host` 的动态 cron；`manifest` cron 仍然必须通过插件自身声明调整
- `POST /api/plugins/remote/bootstrap` 会在线创建或刷新一个 remote plugin 占位记录，并返回专用接入令牌，不需要服务重启

## WebSocket 协议摘要

远程插件和服务器之间的核心动作：

| type | action | 方向 | 说明 |
| --- | --- | --- | --- |
| `auth` | `authenticate` | 插件 -> 服务器 | 认证 |
| `auth` | `auth_ok` / `auth_fail` | 服务器 -> 插件 | 认证结果 |
| `plugin` | `register` | 插件 -> 服务器 | 注册 manifest |
| `plugin` | `register_ok` | 服务器 -> 插件 | 注册成功 |
| `command` | `execute` | 服务器 -> 插件 | 调用插件工具 |
| `command` | `execute_result` / `execute_error` | 插件 -> 服务器 | 工具执行结果 |
| `plugin` | `hook_invoke` | 服务器 -> 插件 | 调用插件 Hook |
| `plugin` | `hook_result` / `hook_error` | 插件 -> 服务器 | Hook 返回结果 |
| `plugin` | `route_invoke` | 服务器 -> 插件 | 调用插件 Route |
| `plugin` | `route_result` / `route_error` | 插件 -> 服务器 | Route 返回结果 |
| `plugin` | `host_call` | 插件 -> 服务器 | 调用 Host API |
| `plugin` | `host_result` / `host_error` | 服务器 -> 插件 | Host API 返回结果 |
| `heartbeat` | `ping` / `pong` | 双向 | 心跳保活；远程插件 `ping` 会刷新插件最近活跃时间 |

SDK 已自动处理：

- 认证
- manifest 注册
- WebSocket 收发
- Host API 往返
- 心跳
- 自动重连

## AI 工具命名规则

- 内建插件工具：直接暴露工具名，例如 `save_memory`
- 远程插件工具：自动加插件前缀，例如 `pc-my-host__list_directory`

这样内建插件可以保持稳定的人类友好工具名，远程插件则天然避免冲突。
